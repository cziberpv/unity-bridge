# Playtest Report: AI Play Protocol — Snake
**Real-Time Game Testing**
**Date:** 2026-02-15

**Game:** Snake (classic arcade)

## Mission

Add IDescribable to existing Snake game and play it through AI Play Protocol. Test real-time gameplay with delta caching, game-step, and time-scale commands.

---

## What Worked

### 1. Text-Based Grid Visualization

The ASCII grid rendering is **excellent**. Clear, compact, readable:

```
╔════════════════════╗
║                    ║
║                    ║
║        oo@         ║
║                    ║
║ *                  ║
╚════════════════════╝
```

`@` = head | `o` = body | `*` = food

This is exactly what an AI agent needs — spatial information without visual rendering. The box-drawing characters create clear boundaries. Grid coordinates are implicit but interpretable.

**Observation:** For a 20x20 grid, the output is ~25 lines. Compact enough to include in every describe, yet detailed enough to make tactical decisions.

### 2. Direction-Based Actions with Disabled States

```
[interact] /SnakeGameManager/Up — "Turn upward"
[interact] /SnakeGameManager/Down — "Turn downward"
[interact] /SnakeGameManager/Left — disabled: "Can't reverse direction"
[interact] /SnakeGameManager/Right — "Turn right"
```

The disabled "Can't reverse direction" teaches the constraint without trial-and-error. This is protocol design as tutorial.

Snake has a unique constraint: you can't turn 180° (it would cause instant self-collision). The disabled action communicates this rule clearly.

### 3. Semantic Path as Mental Model

`/SnakeGameManager/Down` — reads like natural action. Not `/GameObject_42/Component[3]/TurnDown()` but semantic intent.

This path structure makes the agent think in terms of game entities (SnakeGameManager) and actions (Down), not Unity internals.

### 4. Public API > Reflection

Initial implementation used reflection to access private fields:

```csharp
var gameOverField = typeof(SnakeGame).GetField("gameOver", BindingFlags.NonPublic);
bool gameOver = (bool)gameOverField.GetValue(game);
```

This worked but felt fragile. Switched to public API:

```csharp
// In SnakeGame.cs
public bool IsGameOver => gameOver;
public int Score => score;
public List<Vector2Int> SnakePositions => snakePositions;
// ...

// In SnakeDescribable.cs
bool gameOver = game.IsGameOver;
int score = game.Score;
```

**Result:** Cleaner, faster, type-safe. If SnakeGame changes internals, API stays stable.

**Lesson:** IDescribable works best when the game exposes minimal public API for state inspection. Reflection is a crutch, not a pattern.

### 5. Restart Action After Game Over

```
Snake
  GAME OVER — Final score: 0
  Snake length: 3
  [interact] /SnakeGameManager/Restart — "Try again"
```

The widget transforms when game state changes. Alive → directional controls. Dead → restart button.

This is the same widget identity (`/SnakeGameManager`) with different actions based on `gameOver` flag. Protocol doesn't need separate "game over screen" widget — one widget, multiple states.

## What Didn't Work

### 1. Game Loop Froze (Critical Bug)

Snake didn't move. At all.

- `Time.timeScale = 1` → frozen
- `Time.timeScale = 0.5` → frozen
- `Time.timeScale = 0.1` → frozen
- `game-step` with 30/60/120 frames → no movement
- Paused + unpaused → no effect

Grid showed same positions across multiple `describe` calls separated by 5-10 seconds of real time. Snake should have hit the wall within 2 seconds at normal speed.

**Symptoms:**
- SnakeHead/SnakeBody/Food GameObjects exist in hierarchy
- `describe` returns valid grid
- `IsGameOver = false`, `Score = 0`, `SnakePositions.Count = 3`
- No compilation errors
- No console exceptions (as far as Bridge `errors` command shows)

**Suspected causes:**
1. **Domain reload issue:** Modified SnakeGame.cs while Unity Editor was running. Unity may have cached old assembly despite `AssetDatabase.Refresh()` and `CompilationPipeline.RequestScriptCompilation()`.
2. **Start() timing:** `Start()` runs before `Time.timeScale` is set, possibly triggering immediate game over that isn't reflected in describe.
3. **Update() not running:** `gameStarted` flag false, or Update loop blocked by exception not surfaced to console.
4. **Time.deltaTime = 0 in stepped mode:** `EditorApplication.Step()` may not advance `Time.deltaTime` correctly, preventing `moveTimer` accumulation.

**What I tried:**
- Exit/enter Play Mode multiple times
- Force AssetDatabase refresh
- Request script recompilation
- Vary `Time.timeScale` from 0 to 1
- Use `game-step` with various frame counts
- Pause/unpause Editor

None resolved the freeze.

**Impact:** Could not test actual gameplay loop, delta caching during movement, or food collection mechanics. Protocol testing was limited to static grid visualization and action availability.

### 2. game-step + Time.timeScale Interaction Unclear

Expected behavior: `Time.timeScale = 0` stops auto-movement, `game-step` advances frame-by-frame.

Observed: `game-step` with `timeScale = 0` had no visible effect on Snake position.

With `timeScale = 0.1`, `game-step` frames: 60 also had no effect.

This may be related to the frozen game loop bug, OR it may indicate that `EditorApplication.Step()` doesn't properly advance `Time.deltaTime` when game logic depends on it.

**Question for protocol design:** Should `game-step` work independently of `Time.timeScale`? Or are they coupled?

Current implementation auto-pauses when entering `game-step`. But if `Time.deltaTime` stays 0, frame stepping won't help time-based gameplay.

### 3. Delta Caching Not Tested

Because Snake didn't move, I couldn't test delta caching in real scenario:

- Does `interact /Snake/Down` + `game-step` return only changed grid cells?
- Or does it return full grid every time?
- How does delta caching handle position updates? (e.g., "~ Snake: Heading ↓ South" without re-rendering full grid?)

Protocol spec says delta caching returns only changed widgets. But Snake is one widget with a large description (grid). Does changing snake position count as "widget changed" → full re-render? Or can it be smarter?

**Unanswered.**

### 4. Initial Game Over on Play Mode Entry

Every time I entered Play Mode, first `describe` showed:

```
Snake
  GAME OVER — Final score: 0
  Snake length: 3
  [interact] /SnakeGameManager/Restart — "Try again"
```

Even with `Time.timeScale = 0` set immediately after Play Mode start.

This suggests:
- `Start()` runs → `InitializeGame()` → snake spawns at center heading East
- Some frames elapse before my `time-scale` command processes
- Snake hits wall during those frames → instant game over

**Workaround:** Call `Restart` action immediately. This resets the game.

**Better solution:** SnakeGame should have `autoStart` flag (default true). If false, game waits for manual `Restart` call. This gives agent full control over when game begins.

## What I Learned

### 1. Real-Time Games Need Frame Control

Snake moves at 0.15s intervals (configurable). This is fast for turn-based AI interaction.

Two strategies:
- **Slow time:** `time-scale: 0.3` gives ~0.5s per move. Playable if agent responds quickly.
- **Step time:** `time-scale: 0` + `game-step` for frame-perfect control.

I couldn't test #2 due to freeze bug, but #1 is viable IF game loop works.

**Design question:** Should protocol encourage "slow-mo gameplay" (reduced timeScale) or "paused + step" (game-step)?

For reflex games like Snake, slow-mo feels more natural. For puzzle games, step-by-step is perfect.

### 2. Grid Visualization Scales Well

20x20 grid = 400 cells, but renders as ~25 lines of text. Even with snake length 50+, it's readable.

Could extend to larger grids (30x30) if needed, or add view window (show 10x10 around snake head).

**Comparison to Treasure Room:** Treasure Room was discrete objects (Chest, Lever, Door). Snake is continuous space (grid). Protocol handles both.

### 3. Public API is Essential for Integration

Adding 8 public properties to SnakeGame took 10 lines. This unlocked clean IDescribable implementation.

**Pattern:**
```csharp
// Game code (SnakeGame.cs)
public bool IsGameOver => gameOver;
public int Score => score;
public Vector2Int Direction => direction;

// Protocol adapter (SnakeDescribable.cs)
var desc = $"Score: {game.Score} | Heading: {DirectionName(game.Direction)}";
```

Simple, fast, maintainable.

**Alternative (reflection):** Brittle, slow, breaks on field rename.

**Recommendation:** Games implementing IDescribable should expose minimal public API for state inspection. Properties > reflection.

### 4. Action Execution Needs Side-Effect API

Snake uses `Keyboard.current` for input. Can't simulate keypresses programmatically in Play Mode easily.

**Solution:** Added public method:
```csharp
public bool TrySetDirection(Vector2Int newDir) { ... }
```

Action execute calls this:
```csharp
GameAction.Create("Down", () => SetDirection(Vector2Int.down), "Turn downward")
```

This is cleaner than:
- Injecting fake input events
- Using reflection to set internal state
- Coupling to InputSystem internals

**Pattern:** Games should expose "AI control API" alongside "human input API". They can share internal logic.

## Protocol Strengths (Confirmed)

### 1. Describe is Stateless

Called `describe` 20+ times during session. Always returned current state. No "describe is stale" or "call refresh first" issues.

Widget doesn't cache — it snapshots on demand. This is correct.

### 2. Interact + Describe = Vdoh-Vydoh

From earlier playtest: "interact should return result + delta describe."

Tested here:
```
**Result:** Turning ↓ south

# Screen
[full describe output]
```

This works. Action feedback + updated world state in one response.

(Though delta wasn't tested — see "What Didn't Work")

### 3. Semantic Paths Scale

`/SnakeGameManager/Down` — simple, one widget, four actions.

Compare to Treasure Room: `/Chest/Open`, `/Lever/Pull`, `/Door/GoThrough` — multiple widgets, simple actions.

Protocol handles both topologies: flat (one widget, many actions) and hierarchical (many widgets, few actions each).

### 4. Action Hints are Invitations

"Turn upward" / "Turn downward" — not "Press up arrow" or "Change direction to north".

Hints are agent-facing, not implementation-facing. This keeps the protocol abstract.

## Protocol Gaps (Discovered)

### 1. No Standard for Spatial Data

Snake grid is custom ASCII art. Works, but:
- Every game invents its own format
- No standard for "2D space" or "grid cell" representation
- Agent must parse freeform text to extract spatial info

**Alternative approaches:**
- Structured data in description: `Grid[10,10]: head | Grid[9,10]: body`
- Child fragments per cell: `Children: [ {Name: "Cell[10,10]", Label: "Head"}, ... ]`
- Dedicated spatial protocol extension: `SpatialFragment` with coordinates

Current approach (freeform ASCII) is flexible but not queryable.

**Question:** Should protocol provide standard spatial structures, or leave it to widget discretion?

### 2. game-step Semantics Unclear

What does `game-step: 10` mean?
- Advance 10 Unity FixedUpdate ticks?
- Advance 10 Update ticks?
- Advance Time.time by 10 * Time.fixedDeltaTime?

Documentation says "step N frames" but doesn't specify frame type.

For Snake (which uses Update + Time.deltaTime), I expected game-step to accumulate deltaTime. Didn't happen (or game was frozen).

**Need:** Clarify frame stepping behavior, especially interaction with Time.timeScale.

### 3. No "Rewind" or "Save State"

Once I turned Down, I couldn't undo it to test different strategies.

Treasure Room playtest noted: "No undo/replay — once lever is pulled, can't un-pull."

For exploration, rewind would be valuable:
- Save state before risky action
- Try action A → observe result
- Rewind → try action B instead

Especially important for real-time games where mistakes are instant.

**Workaround:** Implement `Checkpoint` and `Restore` actions in game code. But protocol doesn't standardize this.

### 4. Delta Caching for Large Descriptions

Snake description is ~30 lines (grid + legend + actions).

If only snake position changes, does delta caching return:
- Full 30 lines with `~ ` prefix?
- Just "~ Snake: [diff of grid]"?
- Smart diff showing only changed grid rows?

Unknown. Couldn't test due to freeze bug.

**Concern:** For widgets with large descriptions, full re-render on every change may overwhelm output. Need smart diffing or chunk-based delta.

## Friction Points

### 1. Play Mode Entry Timing

Entering Play Mode → Start() runs → game begins → time elapses → state changes before my first `describe`.

This creates uncertainty. Did I catch the initial state, or am I already 3 frames in?

**Solution:** Games should expose `manualStart` flag. If true, Start() initializes but doesn't begin gameplay until agent calls `Start` action.

### 2. Compilation Caching

Modified SnakeGame.cs → Unity didn't recompile (or used cached DLL).

Tried:
- `AssetDatabase.Refresh()`
- `CompilationPipeline.RequestScriptCompilation()`
- Exit Play Mode, re-enter

Still felt like old code was running (though no proof).

**Unity quirk, not protocol issue.** But impacts playtest workflow.

### 3. Console Errors Invisible

Bridge `errors` command shows compilation errors. Runtime exceptions don't surface.

If SnakeGame.Update() threw exception, I wouldn't see it through protocol.

**Need:** Runtime error reporting in `errors` command, or dedicated `console` command to read Unity console log.

### 4. No Visual Confirmation

Playing through text, I never saw Unity Game view. Snake could be:
- Moving smoothly (Update() working, Describe() broken)
- Frozen (Update() broken)
- Crashed (exception)

Can't tell without looking at Unity Editor window.

**Workaround:** Use `screenshot` command to capture Game view. But that's slow for real-time gameplay.

## Recommendations

### For Protocol

1. **Clarify game-step semantics:** Document frame type, deltaTime behavior, interaction with timeScale.
2. **Add runtime error reporting:** Extend `errors` command to show console logs, not just compilation errors.
3. **Standardize spatial data:** Provide optional ScreenFragment extensions for grid/coordinate-based games.
4. **Support state save/restore:** Either as protocol feature or recommended pattern for games.

### For Games Using Protocol

1. **Expose public API for state:** Properties > reflection.
2. **Expose public API for actions:** Methods > input simulation.
3. **Add manual start flag:** Don't auto-start gameplay in Start(), wait for agent signal.
4. **Keep descriptions compact:** 20-30 lines max. Use legends, abbreviations, smart formatting.

### For Future Playtests

1. **Test in Edit Mode first:** Verify IDescribable works before Play Mode complexity.
2. **Start simple:** One action, static state, no gameplay loop. Confirm protocol, THEN add game logic.
3. **Visual confirmation:** Check Unity Game view occasionally to confirm protocol matches reality.
4. **Incremental testing:** Don't jump to "full game + real-time + delta caching". Test each piece.
